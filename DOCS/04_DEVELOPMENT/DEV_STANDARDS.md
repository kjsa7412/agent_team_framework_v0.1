# 개발 표준 (Development Standards)

## 기본 원칙

1. 단순함 우선 — 복잡한 추상화보다 단순하고 명확한 코드
2. 일관성 — 팀 전체가 같은 패턴 사용
3. 가독성 — 코드는 문서다

## 브랜치 전략

```
main         # 운영 브랜치
  └─ feature/{WI-ID}-{설명}  # 기능 브랜치
  └─ fix/{WI-ID}-{설명}      # 버그 수정 브랜치
  └─ hotfix/{설명}           # 긴급 수정 브랜치
```

## 커밋 메시지

```
[WI-ID] 유형: 내용

유형:
- feat: 새 기능
- fix: 버그 수정
- refactor: 리팩토링
- docs: 문서
- test: 테스트
- chore: 빌드/설정

예: [WI-20260311-001] feat: 사용자 로그인 API 구현
```

## 코드 리뷰

- 모든 변경은 PR을 통해 병합
- 최소 1명 리뷰 후 머지
- 자동화 테스트 통과 후 머지

## 금지 사항

- 하드코딩된 시크릿
- console.log 운영 코드 잔류
- TODO 주석 미해결 상태로 머지
- 테스트 없는 핵심 기능 구현
