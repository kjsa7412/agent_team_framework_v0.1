# 테스트 정책 (Test Policy)

## 테스트 조직 독립 원칙

Test Group은 Foundation/Evolution의 하위 역할이 아닌 독립 조직으로 운영한다.

이유:
- 구축/운영/구조 재편 모두를 검증해야 함
- 특정 구현 그룹에 종속되면 검증 관점이 약화됨
- 테스트 자산은 프로젝트 전반에 재사용되어야 함

## 테스트 케이스 정의

테스트 케이스 = 원자적 검증 단위

예:
- 로그인 성공
- 로그인 실패 (잘못된 비밀번호)
- 필수값 누락 시 에러 표시
- 권한 없는 접근 차단

## 테스트 시나리오 정의

테스트 시나리오 = 테스트 케이스의 조합

변경분 크기와 영향도에 따라 케이스를 선택하여 구성.

## 변경 규모별 테스트 수준

### evolution-small
- 최소 영향 범위 시나리오
- 변경 기능 직접 관련 케이스 우선
- 핵심 회귀 케이스 일부 포함

### evolution-large
- 변경 기능군 전체 케이스 조합
- 영향받는 인접 기능 회귀 포함
- 인증/권한/상태변이 관련 케이스 확대

### foundation-rebuild
- 핵심 플로우 전면 시나리오
- 기반 기능 재검증
- 공통 구조, 권한, 예외 처리, 배포 후 스모크 포함

### hybrid-foundation-evolution
- Foundation 재정의 영향 케이스
- Evolution 반영 기능 케이스
- 양쪽 통합 시나리오

## 실행 원칙

1. 테스트 케이스는 기능 단위로 축적 관리
2. 시나리오는 변경분 크기 기준으로 케이스를 조합하여 구성
3. Test Group은 무엇을 테스트할지 스스로 설계
4. 구현 그룹은 테스트 입력(변경 내역, 영향 범위)을 제공
